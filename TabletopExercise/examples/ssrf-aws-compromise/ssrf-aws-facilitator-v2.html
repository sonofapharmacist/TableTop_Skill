01

Executive Summary

This tabletop exercise simulates a critical security incident where a threat actor exploits a Server-Side Request Forgery (SSRF) vulnerability in the organization's e-commerce web application to access the AWS EC2 instance metadata service.

The attacker successfully steals temporary IAM role credentials and uses them to access sensitive S3 buckets containing customer PII. The scenario tests the organization's ability to detect SSRF exploitation, respond to AWS credential compromise, contain the breach, and implement cloud security controls.

This is NOT a test of individual performance. The goal is to evaluate organizational procedures, identify gaps, and improve response capabilities.

02

Scenario Overview

Your organization runs a Flask-based e-commerce platform on AWS EC2. The application includes a product price lookup feature that fetches pricing from external APIs. A developer recently introduced a feature that allows users to submit URLs for the application to fetch data from - but the implementation lacks any input validation.

Your SOC is monitoring network traffic when an alert fires: unusual outbound HTTP requests from your production web server to IP address 169.254.169.254 - the AWS EC2 Instance Metadata Service. This is a reserved IP that should never be accessible from the internet, yet your server is making hundreds of requests to it.

Simultaneously, customer support receives reports that customers are receiving password reset emails they didn't request.

SSRF-DETECT-001 T+0 Initial Detection high

Scenario

Your SOC monitoring dashboard fires a high-severity alert: unusual outbound HTTP traffic to 169.254.169.254 (AWS EC2 Metadata). 247 requests from production web server over 30 minutes. Simultaneously, 12 customers report unauthorized password reset emails.

Artifact

SECURITY ALERT - HIGH SEVERITY
Alert: SSRF-001 - Suspicious Metadata Access
Source: ec2-prod-web-01 (10.0.15.42)
Destination: 169.254.169.254 (AWS Metadata)
Request Count: 247 requests in 30 minutes

Request Pattern:
  /latest/meta-data/
  /latest/meta-data/iam/
  /latest/meta-data/iam/security-credentials/
  /latest/meta-data/iam/security-credentials/ec2-prod-role

Associated: 12 customer tickets - unauthorized password resets

Expected Response

Assign incident commander, isolate affected server, preserve evidence, begin forensic analysis, escalate to management, connect customer reports to attack.

SSRF-CONTAIN-001 T+40 Containment - Persistence critical

Scenario

Attacker created IAM user 'aws-admin-backup' and Lambda 'data-sync-handler' for persistence. Modified legitimate Lambda 'order-processor' to exfiltrate new data. Attacker establishing backdoor.

Artifact

CloudTrail - Persistence Detected:
+ Created IAM user: aws-admin-backup
+ Created Lambda: data-sync-handler (exfil code)
~ Modified Lambda: order-processor (added data exfil)

Expected Response

Delete malicious resources, rotate credentials again, full audit of IAM/Lambda, preserve evidence before cleanup.

SSRF Incident Response Playbook

Evaluation Trigger: Initial detection of SSRF attack targeting metadata service

Required Procedures

  • SSRF-specific detection and classification procedures
  • Incident command structure during SSRF incidents
  • Step-by-step containment (isolate, preserve evidence, investigate)
  • Escalation thresholds and decision trees
  • AWS credential rotation procedures

Impact if Gap Remains

Delayed response to SSRF attacks, confusion about roles, potential for credential theft to go undetected longer, regulatory penalties for slow breach notification

Recommendation

Develop SSRF-specific incident response playbook including metadata service blocking procedures, automated detection rules, and credential rotation runbooks. Test quarterly through tabletop exercises.

CRITICAL

AWS Credential Rotation Procedure

Evaluation Trigger: Discovery of compromised AWS credentials

Required Procedures

  • Emergency credential rotation runbook
  • IAM role vs. user credential procedures
  • Automated credential rotation
  • Verification procedures post-rotation

Recommendation

Document emergency credential rotation steps, create runbook for IAM role vs. user credentials, implement automated rotation where possible.

HIGH

EC2 IMDS Configuration Standards

Evaluation Trigger: Post-incident hardening review

Required Procedures

  • IMDSv2 enforcement on all new instances
  • Security group rules blocking 169.254.0.0/16
  • Instance profile least-privilege
  • Regular audit of metadata access

Recommendation

Enable IMDSv2 on all instances, implement security group egress rules blocking metadata IP ranges, apply least-privilege IAM roles.
