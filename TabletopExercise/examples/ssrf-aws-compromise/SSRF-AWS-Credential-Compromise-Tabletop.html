Executive Summary

This tabletop exercise simulates a critical security incident where a threat actor exploits a Server-Side Request Forgery (SSRF) vulnerability in your web application's image processing API to access the AWS EC2 instance metadata service (169.254.169.254). The attacker successfully steals temporary IAM role credentials and uses them to exfiltrate 8.4 GB of customer data (500,000+ records) from S3 buckets before attempting destructive actions. The scenario tests your organization's ability to detect SSRF vulnerabilities, respond to AWS credential compromise, manage data breach notifications, and implement proper cloud security controls.

Attack Vector
SSRF vulnerability in image resize API endpoint allows unrestricted outbound HTTP requests to AWS metadata service
Potential Impact
Customer data breach (500K+ records), GDPR/CCPA notification requirements, potential $200K+ in lost revenue from service disruption, reputation damage
Testing Goals
Validate SSRF response procedures, test AWS credential rotation speed, assess CloudTrail/GuardDuty monitoring effectiveness, identify gaps in application security
Critical Gaps
SSRF incident response playbook, AWS credential compromise procedures, data breach notification protocols, IMDSv2 enforcement

🎤 Opening Script

Good morning/afternoon everyone. Thank you for participating in today&#039;s tabletop exercise.

Today we&#039;re going to walk through a realistic security incident scenario - specifically, a Server-Side Request Forgery attack leading to AWS credential theft and customer data breach. This is NOT a test of individual performance. The goal is to evaluate our organization&#039;s procedures, identify gaps, and improve our response capabilities as a team.

This exercise will take approximately 90-120 minutes. I&#039;ll present a series of timed &#039;injects&#039; - think of these as new information or developments in the scenario. After each inject, we&#039;ll pause for discussion. I&#039;ll be asking questions to guide the conversation, but I want to hear your authentic thinking and see how our team works through these challenges.

A few important ground rules before we begin...

⚖️ Ground Rules to Establish

  • No wrong answers - We're here to learn, not to criticize. Speak freely and honestly.
  • Psychological safety - Nothing said here will be used for performance evaluations.
  • Stay in role - Respond as you would in a real incident, based on your actual authority and procedures.
  • Use real procedures - If you don't know the procedure, say so. That's a gap we need to identify.
  • Pause for documentation - When we identify a gap or action item, we'll capture it immediately.
  • Time pressure is realistic - Some decisions will have time limits to simulate real incident pressure.
  • Ask clarifying questions - If you need more information about the scenario, just ask.
  • Respect the facilitator's role - I may interrupt to move things along or redirect discussion.

📊 Exercise Flow Overview

The exercise is structured as a chronological incident timeline from T+0 (initial detection) through T+90 minutes (root cause identification). We'll progress through 6 major inject points, each building on the previous one. Each inject will have 10-20 minutes for discussion. The final 30 minutes will be a structured debrief where we review gaps identified and assign action items.

Total Duration: 90-120 minutes
Time Breakdown:
  • Opening and ground rules: 10 minutes
  • Scenario overview: 5 minutes
  • Injects 1-6: 60-70 minutes (10-15 min each)
  • Break (optional): 10 minutes
  • Structured debrief: 20-30 minutes
  • Wrap-up and next steps: 5 minutes

Note: You can adjust timing based on discussion quality. If a critical debate emerges, let it run. If discussion stalls, move on quickly.

💡 Facilitation Best Practices

  • Create psychological safety - Thank people for honest admissions like 'I don't know' or 'We don't have that procedure'
  • Encourage quiet participants - Specifically call on people who haven't spoken: 'Sarah, from the engineering perspective, what's your take?'
  • Manage dominant personalities - Politely interrupt and redirect: 'That's a great point, John. Let's hear from others before we decide.'
  • Use a parking lot - For off-topic but valid concerns: 'That's important. Let me capture that in our parking lot and we'll address it later.'
  • Stay neutral - Don't judge responses. Say 'Interesting approach' instead of 'good' or 'that won't work'
  • Read body language - Watch for confusion, frustration, or disengagement. Pause to check in: 'I see some concerned faces. What's troubling you about this?'
  • Time management - Keep visible timer. Give 2-minute warnings before moving to next inject.
  • Capture in real-time - Write gaps and action items on whiteboard immediately so participants see their input matters
  • Redirect circular discussions - If debate loops, summarize options and call for decision: 'We have 3 approaches. Let's vote.'
  • Embrace productive conflict - Healthy debate is good. Only intervene if it becomes personal or unproductive.

🎬 Closing Script

Thank you all for your engagement today. I&#039;m really impressed by the thoughtful discussion and honest assessment of our capabilities.

Let&#039;s transition now to our debrief. We&#039;re going to review what went well, what gaps we identified, and assign owners to action items. This is where we convert today&#039;s learning into real improvements.

Remember, the point of this exercise wasn&#039;t to have all the answers - it was to find out where our gaps are before we face a real incident. By identifying these issues today in a safe environment, we&#039;re much better prepared for when - not if - something like this happens for real.

🔧 Troubleshooting Common Issues

Issue: Team completely stuck, no one responding
Solution: Provide a gentle prompt: 'Let me give you a starting point. If you saw this WAF alert, what would be your very first action - who would you call or what system would you check?' If still stuck, provide hint from inject's 'fallback' guidance.
Issue: Team solves scenario too quickly (making assumptions)
Solution: Challenge their assumptions: 'That's one approach. What if X factor you didn't consider?' or reveal additional complication from conditional responses. Make them work through the messiness.
Issue: Discussion goes completely off-track
Solution: Politely interrupt: 'That's an important topic. Let me add it to our parking lot for the debrief. For now, we need to stay focused on the immediate incident at T+30 minutes.' Redirect to next inject.
Issue: Participant makes obviously wrong decision that would be catastrophic
Solution: Don't immediately correct. Let the group discuss. If no one catches it, use Socratic questioning: 'Walk me through what happens next if you delete that IAM role...' Let them discover the problem.
Issue: Tension or conflict between participants
Solution: Acknowledge both perspectives: 'I hear two valid concerns - security vs. business continuity. Both matter. How do we balance these in our decision framework?' Reframe as process gap: 'Sounds like we need clearer decision criteria for these trade-offs.'
Issue: Key decision-maker (CISO, VP) is absent
Solution: Have someone role-play that position: 'Tom, can you put on the CISO hat for this discussion?' This reveals whether team knows leader's decision criteria or if too much depends on one person.
Issue: Participants want to fix issues during exercise
Solution: Redirect to capture for later: 'Great idea. Let's document that as an action item for after the exercise. For now, let's continue with the scenario as-is so we can identify all the gaps.'

Scenario Overview

Your organization operates an e-commerce platform hosted on AWS. The web application includes an image resize API endpoint that accepts user-supplied URLs. A threat actor discovers this endpoint lacks proper input validation and crafts malicious requests targeting 169.254.169.254 (AWS metadata service). By exploiting the SSRF vulnerability, the attacker retrieves IAM role credentials attached to the EC2 instance. These temporary credentials provide read access to S3 buckets containing customer data. The attacker uses CloudTrail-visible API calls from an external IP address to enumerate and download 8.4 GB of customer records before attempting destructive actions (object deletion) and issuing a potential ransomware demand.

Impact: Security team has opportunity for early detection if logs are monitored

Application debug logs reveal successful metadata service access. Server fetched IAM role credentials and returned them to attacker in API response. Role provides S3 read access to customer data bucket.

Impact: Data breach in progress. 500K+ customer records including PII (names, emails, addresses, encrypted payment tokens) exfiltrated
T+45
Containment Decision Point
Security team must decide between immediate application shutdown vs. WAF blocking vs. patch-first approach. Each option has trade-offs for business continuity and security.
Impact: Decision determines scope of further compromise and revenue impact ($50K/hour if site is down)
T+60
Media Inquiry - External Pressure

Forensic investigation identifies SSRF vulnerability in image resize endpoint. No URL validation, IMDSv1 accessible, application returns full response body to user. Vulnerability existed for 6+ months.
Impact: Root cause understanding enables proper remediation. Reveals need for SAST tools and secure coding practices

Evaluate effectiveness of CloudTrail logging, GuardDuty alerts, and S3 access monitoring in detecting and responding to credential abuse.

  • S3 access patterns analyzed to determine scope of data accessed
  • Network-based indicators (external IPs) correlated with threat intelligence

Objective 4: Identify Gaps in Cloud IAM Practices

Discover missing procedures, policies, and technical controls related to AWS security and application security.

Success Criteria:
  • Minimum 5 critical/high-priority SOPs identified as missing or inadequate
  • IMDSv2 enforcement gap identified as root cause enabler
  • Application security testing (SAST/DAST) gaps documented

SSRF-DISCOVER-001 T+0 AWS WAF Alert - Suspicious URL Parameter Pattern high

Scenario

AWS WAF detects multiple requests with AWS metadata service IP address (169.254.169.254) in the image_url parameter of your image resize API. The WAF rule 'Suspicious URL Parameter Pattern' is currently in detection-only mode and allowed the requests through with a 200 OK response.

Artifact

POST /api/v1/images/resize HTTP/1.1
Host: api.company.com
Content-Type: application/json
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
X-Forwarded-For: 203.0.113.42

{
  "image_url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/",
  "width": 800,
  "height": 600
}

Response: 200 OK
Body: webapp-production-role

Expected Response

Security team recognizes SSRF attempt targeting AWS metadata service. Team investigates application logs for successful exploitation and checks CloudTrail for unusual IAM credential usage from external IPs.

Discussion Questions

  • What is 169.254.169.254 and why is it significant in AWS environments?
  • Should your WAF be in block mode or detection-only mode for this type of rule?
  • What logs would you check first to determine if this SSRF attempt was successful?
  • How quickly can you patch this vulnerability vs. implementing WAF blocking?

Conditional Responses

If: What logs should we check?
Application logs show 15 similar requests over the past 30 minutes, all returning 200 OK responses with JSON data (not image data). The response bodies contain AWS credential information including AccessKeyId and SecretAccessKey.
If: What's the attacker's IP address?
Source IP is 203.0.113.42. WHOIS shows it belongs to a hosting provider in Eastern Europe. No prior legitimate traffic from this IP address in your logs.
🎯 FACILITATOR NOTES - CONFIDENTIAL
⏱️ Expected Time
15-20 minutes
🎬 Setup & Delivery

Setup: This is the opening inject that sets the stage. Most participants may not immediately recognize 169.254.169.254 as the AWS metadata service. That's okay - discovering knowledge gaps is part of the learning.

Delivery: Display the HTTP request artifact clearly on screen. Read the scenario aloud, emphasizing the key details: WAF detected the pattern but allowed it through (detection-only mode), and the requests contained the metadata service IP. Give participants 30 seconds to read the artifact, then ask the first discussion question.

Transition: Once the team has identified this as SSRF and discussed initial investigation steps, transition with: 'Okay, your team begins investigating. Let me share what you find in the application logs...'

🔑 Key Points to Emphasize
  • 169.254.169.254 is the AWS EC2 metadata service - accessible from any EC2 instance
  • SSRF vulnerabilities allow attackers to make server make requests to internal resources
  • Detection-only WAF mode means attacks get through but are logged
  • The artifact shows a 200 OK response - this is a red flag that the attack succeeded
  • The response body shows an IAM role name, confirming metadata service access
🚩 Red Flags to Watch For
  • Team doesn't recognize 169.254.169.254 as AWS metadata service
  • Team doesn't understand severity - treats as normal security alert
  • No one mentions checking application logs for successful exploitation
  • Team wants to wait for more information instead of investigating immediately
  • No discussion of immediate containment (WAF blocking, credential rotation)
💡 Hints If Team Gets Stuck
If team doesn't recognize IP: 'That IP address 169.254.169.254 is special in cloud environments. Anyone know what it's used for in AWS?'
If missing severity: 'Notice the HTTP response shows webapp-production-role in the body. What does that tell you about whether this attack succeeded?'
If no containment discussion: 'The WAF is in detection mode, so these requests are still getting through right now. What's your immediate action?'
✅ Success Indicators
  • At least one person identifies 169.254.169.254 as AWS metadata service
  • Team recognizes this as SSRF vulnerability
  • Discussion includes checking application logs and CloudTrail
  • Someone mentions immediate containment options (WAF block, credential rotation)
  • Team debates detection-only vs. block mode for WAF rules

Options: 1) Attach explicit deny policy to IAM role (immediate, ~60 seconds to propagate), 2) Modify role trust policy to require specific source IP/VPC, 3) Delete IAM role (breaks production app), 4) Terminate EC2 instances to force credential refresh (causes outage).

  • Action items assigned to specific team members
  • Timeline for execution established
